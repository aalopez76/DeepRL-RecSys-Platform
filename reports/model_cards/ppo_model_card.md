## 1. Model Description

### Algorithm
**Proximal Policy Optimization (PPO)** — an on-policy policy gradient algorithm that uses a clipped surrogate objective to enforce conservative, stable policy updates. PPO alternates between collecting experience and optimizing the policy for multiple epochs per batch.

### Architecture

| Component | Architecture |
|-----------|-------------|
| **Shared backbone** | 2-layer MLP: `input_dim → 128 → 128`, Tanh activations |
| **Policy head** | Linear: `128 → n_actions`, softmax output |
| **Value head** | Linear: `128 → 1` (for GAE advantage estimation) |

### Hyperparameters

| Parameter | Value |
|-----------|-------|
| Learning rate | 3e-4 |
| Discount factor (γ) | 0.99 |
| Clip ratio (ε) | 0.2 |
| Optimization epochs per batch | 4 |
| Batch size | 64 |
| GAE lambda (λ) | 0.95 |
| Seed | 42 |

## 2. Training Data

| Scenario | Source | Size |
|----------|--------|------|
| **Synthetic** | Generated by `SyntheticBanditEnvironment` | 5,000 samples, 50 items |
| **OBD Random** | Open Bandit Dataset (Uniform logging policy) | 5,000 samples, 80 items |
| **OBD BTS** | Open Bandit Dataset (BTS logging policy) | 5,000 samples, 80 items |

> **Note on PPO with offline data:** PPO is fundamentally an on-policy algorithm. Using offline (bandit feedback) data for PPO training is a known mismatch — the algorithm is adapted here as a functional stub to demonstrate the framework's multi-agent support. For production, on-policy simulation or importance sampling corrections should be applied.

## 3. Performance Metrics

### OPE Estimates (IPS / DR / MIPS)

| Scenario | IPS | DR | MIPS | ESS | Reliability |
|----------|-----|----|------|-----|-------------|
| Synthetic | 0.01061 | 0.01061 | 0.01221 | ~3,723 | ✅ Reliable |
| OBD Random | 0.01061 | 0.01061 | 0.01221 | 3,723 | ✅ Reliable |
| OBD BTS | ~0.00488 | ~0.00488 | ~0.01397 | ~2,029 | ⚠️ Warning |

> **Note:** PPO OPE values are **identical to DQN** across all scenarios. Both agents output near-random action distributions and are statistically indistinguishable (Wilcoxon p ≈ 1.0). This is expected behavior for untuned stub agents on batch offline data.

### Context Sensitivity
- Spearman ρ (perturbation test): **~0.99** (context-insensitive)
- Top-5 overlap after context perturbation: **~100%**
- PPO does **not** meaningfully personalize recommendations in current form ❌

---

## 4. Limitations

1. **Stub implementation:** PPO has not undergone extended training or hyperparameter optimization. Current results are near-initialization baselines.
2. **On-policy / offline mismatch:** PPO is designed for online interaction; applying it to static bandit feedback data requires adaptation (e.g., importance sampling), which is not yet implemented.
3. **Context-agnostic:** Like DQN, PPO's current policy does not differentiate rankings based on user affinity context.
4. **Identical to DQN on OBD:** Both produce the same OPE estimates and importance weight distributions — statistical significance tests confirm p ≈ 1.00. This is correct behavior for untrained stubs.

## 5. How to Use

### Load from Artifacts

```python
from deeprl_recsys.serving.runtime import ServingRuntime

runtime = ServingRuntime(model_dir="artifacts/models/benchmark_ppo_random")
context = {"user_item_affinity": 0.3, "user_id": 55}
candidates = list(range(80))

recommendations = runtime.predict(context, candidates, k=5)
```

### CLI Evaluation

```bash
deeprl-recsys evaluate --config configs/experiments/exp_ppo_random.yaml
```

### Full Benchmark

```bash
python scripts/run_full_benchmark.py --agent ppo
```
